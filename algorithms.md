# weiqi3 算法与优化全览

从项目启动到当前状态，所有使用过、尝试过、弃用过的算法和优化手段。

---

## 一、搜索算法

### 1. Minimax（极小化极大）⛔ 已弃用

**描述**：零和博弈经典算法。Max 层取最大值，Min 层取最小值，交替递归。

**为什么用**：用户最初要求"用 minimax，不要 MCTS，因为落子点 < 30"。

**为什么弃用**：用户后续要求"不能容忍错误，必须得到唯一最优解"。minimax 依赖静态评估函数打分，**在截断深度上的评分是近似值而非真相**，无法提供数学上的严格最优保证。

---

### 2. Negamax（取反极大）⛔ 已弃用（随 minimax 一起）

**描述**：minimax 的等价简化写法。利用零和性质 `min(a,b) = -max(-a,-b)`，用一份代码 + 取反替代 Max/Min 两套逻辑。

**为什么用**：代码量减半，逻辑更清晰。

**为什么弃用**：与 minimax 一起被 df-pn 替代。

---

### 3. Alpha-Beta 剪枝 ⛔ 已弃用（随 minimax 一起）

**描述**：在 minimax/negamax 搜索中维护 [α, β] 窗口。当某子节点的值超出窗口，说明该分支不会被选中，立即停止搜索（β 剪枝）。最好情况下搜索量从 b^d 降到 b^(d/2)。

**为什么用**：minimax 的标准配套加速，效果依赖走法排序质量。

**为什么弃用**：整个 minimax 框架被替换。

---

### 4. df-pn（Depth-First Proof Number Search）✅ 当前使用

**描述**：Kishimoto 2002 提出的证明数搜索。维护每个节点的 pn（证明攻方胜的代价）和 dn（证明防方胜的代价）。OR 节点 pn=min(子.pn), dn=sum(子.dn)；AND 节点反之。每次选最小 pn（OR）或最小 dn（AND）的子节点展开。

**为什么用**：
- 不依赖静态评估函数（终止条件是精确的"被提子"或"双眼"）
- 搜索完成时结论数学上严格
- 搜不完时返回 UNPROVEN（绝不猜测）
- 对死活题是业界标准算法

**局限**：搜索空间随分支因子指数增长，空位 > 15 时难以在合理时间内收敛。

---

### 5. MCTS（蒙特卡洛树搜索）⛔ 从未使用

**描述**：通过随机模拟（rollout）估计每个走法的胜率，逐步构建非对称搜索树。AlphaGo 的核心算法。

**为什么没用**：用户明确要求不使用 MCTS，理由是"落子点不会太多"。MCTS 的优势在高分支因子（全盘 19×19）；对 10×10 局部死活题，精确搜索（minimax/df-pn）更适合。

---

## 二、评估与终止

### 6. 启发式评估函数 ⛔ 已弃用

**描述**：对棋局打分的加权公式。特征包括：真眼数、气差、做眼空间、棋子数等，按死活/对杀类型切换权重。

**为什么用**：minimax 在截断深度上需要一个"局面好坏"的近似值。

**为什么弃用**：启发式评估是近似而非精确。df-pn 用精确终止条件（被提子 / 双眼）替代，不再需要启发式评估。

---

### 7. 死活/对杀分类器 ⛔ 已弃用

**描述**：`classifyProblem()` 根据弱群数量、共享气等特征判断当前局面是"死活题"还是"对杀题"，切换评估函数的权重。

**为什么用**：minimax 的评估函数需要针对不同题型调权。

**为什么弃用**：df-pn 不区分题型——统一的终止条件（提子/双眼）适用于所有情况。

---

### 8. 严格真眼判定 ✅ 当前使用

**描述**：判断空点是否为特定群的"真眼"。条件：①空点 ②4 个正交邻全属同一连通群 ③对角规则（内部 ≥3 同色，边角全部同色）。

**为什么用**：df-pn 的终止条件之一——目标群有 ≥2 真眼即判防方胜。真眼判定必须精确，否则终止判定会出错。

---

### 9. 复合终止条件（多目标）✅ 当前使用

**描述**：支持同时指定多个"杀目标"和"守目标"。攻方胜 = 所有杀目标被提；防方胜 = 任一杀目标做出双眼 OR 任一守目标被提。

**为什么用**：用户需要"同时吃两块白棋 + 保住一块黑棋"的复合解题场景。

---

## 三、搜索优化（df-pn 内部）

### 10. 转置表（TT）✅ 当前使用

**描述**：用 dict 缓存每个 `(局面哈希, 行棋方, ko状态)` 的 pn/dn 值。相同局面经不同路径到达时可直接查表，避免重复搜索。

**为什么用**：围棋搜索中同一局面可能经多条路径到达，TT 避免指数级重复。

---

### 11. 跨请求 TT 缓存 ✅ 当前使用

**描述**：模块级 `_TT_CACHE`，按 `(region, kill_targets, defend_targets, attacker)` 索引。同一道题的多次 `solve()` 调用共享一张 TT。

**为什么用**：自动对弈连续 9 手都是同一棵证明树的子局面。第一手建树后，后续步骤几乎全部 TT 命中。实测后续 8 手从 2427ms 降到 **7ms**（347× 加速）。

---

### 12. 穷举根证明 ✅ 当前使用

**描述**：df-pn 找到第一个胜着即停止，其他根子节点未被证明。穷举根证明：在主搜索完成后，对所有未决的根子节点继续 `_mid` 直到证完。

**为什么用**：让 `_extract_best_move` 能在所有真胜着中按棋形要点（vitalness）选最优，而不是只返回第一个碰巧被证明的着法。

---

### 13. Vitalness 棋形破平 ✅ 当前使用

**描述**：对已证明的多个胜着按"要点性"排序。vitalness(x,y) = 该点相邻方位中属于根杀目标气的数量。数字越大 = 同时紧到越多目标气 = 真正的"棋形要点"。

**为什么用**：df-pn 的默认破平是"最快证明"（min dn），不感知棋形。多个等效杀着时可能返回非要点。vitalness 让边线板四的中心点优先返回。

---

### 14. 顽抗着（落败方落子）✅ 当前使用

**描述**：当一方已被证明必败，仍选择"对手最难推进"的着法（max 对方 pn/dn）继续下棋，而非 pass 或停止。

**为什么用**：保证自动对弈能完整走到棋盘真正终局（目标被提子或做出双眼），而不是半途"我输了所以不下了"。

---

### 15. 走法排序（capture-first + vitalness + proximity）✅ 当前使用

**描述**：`_gen_children` 中对候选走法评分排序。评分 = 提子加分(+10000) + vitalness(+100) + 邻石(+5)。df-pn 展开按此顺序选子节点。

**为什么用**：好的走法排序让 df-pn 更快收敛——先探索最有可能构成证明的分支。

**注意**：曾尝试加入"叫吃检测"（落子后检查对方邻群气=1），但需要额外 `group_and_libs` 调用，**导致吞吐量降低 4×**（从 67k 降到 16k nodes/s）。权衡后移除叫吃检测，只保留零开销的提子计数。

---

### 16. 迭代加深 ⛔ 已弃用（随 minimax 一起）

**描述**：从浅层开始搜索（depth=2），逐步加深到 4, 6, 8...。每层搜索完成后检查时间预算。

**为什么用**：minimax 的标准配套——浅层搜索的 TT 信息帮助深层搜索的走法排序；任何时刻可中断返回当前最佳。

**为什么弃用**：df-pn 不按固定深度搜索，而是按 pn/dn 驱动，自动聚焦最需要搜索的分支。无需手动控制深度。

---

### 17. 走法分支上限（max 15）⛔ 已弃用

**描述**：minimax 中每层最多考虑 15 个候选走法（排序后截断）。

**为什么用**：控制搜索规模，防止时间爆炸。

**为什么弃用**：分支上限可能丢掉真正的最佳手（如果排序启发式错了）。df-pn 搜索所有合法着，不做截断——保证数学严格性。

---

## 四、棋盘引擎优化（JS 版，v1）

### 18. Zobrist 增量哈希 ✅ JS 版使用 / Python 版未使用

**描述**：预计算随机数表 `ZOBRIST[position][color]`。每次放/提子时 XOR 更新哈希，O(1) 获取局面哈希。

**为什么用（JS）**：原先的 100 字符字符串哈希每次 O(100)。Zobrist 让 TT 查询从 O(100) 降到 O(1)，节省大量字符串分配。

**为什么 Python 版不用**：Python 的 dict 键哈希对字符串已经足够快（C 层实现）。Zobrist 在 Python 中的 XOR 运算反而没有 C 字符串哈希快。可读性优先。

---

### 19. 撤销式落子（play/undo）✅ JS + Python 都使用

**描述**：`play_undoable()` 返回 undo 句柄（记录放了什么、提了什么、旧 ko 状态），`undo()` 原地恢复。搜索时不再 `clone()` 棋盘。

**为什么用**：消除每步一次的 `Int8Array(100)` / `list(100)` 分配 + GC 压力。在 JS 中节省约 20% 时间。

---

### 20. Epoch 计数器替代 Set ✅ JS 版使用

**描述**：模块级 `Int32Array(100) _GAL_VISITED + _GAL_LIB_MARK`，每次调用 `groupAndLibs` 时 `epoch++`。用 `array[i] === epoch` 代替 `Set.has(i)`。重置只需 `epoch++`，不需清空数组。

**为什么用**：JS 的 `Set.add / Set.has` 每次做字符串/数字哈希，比直接数组索引慢 3× 以上。此优化让 `groupAndLibs` 快了约 30%。

**Python 版**：使用 `bytearray(100)` 做 visited 标志（类似思路但更简单），不用 epoch。

---

### 21. 内联 4 邻展开 ✅ JS + Python 都使用

**描述**：把 `for nx, ny in neighbors(x, y):` 生成器调用展开为北/南/西/东 4 个直接 `if` 判断。

**为什么用**：在 Python 和 JS 中，生成器/迭代器有显著开销（函数调用、yield、tuple 创建）。展开后代码更长但零函数调用。在 Python PyPy 上贡献了约 30% 加速。

---

### 22. wantGroup 懒分配 ✅ JS 版使用

**描述**：`groupAndLibs(x, y, wantGroup=false)` 参数。大多数调用（如 play 的捕获检测）只需要 `libs` 计数，不需要 `group: [[x,y],...]` 嵌套数组。`wantGroup=false` 时跳过 group 和 groupMask 的分配。

**为什么用**：V8 在嵌套调用中无法消除 `[x,y]` 小数组分配。140 万次热路径调用 × 每次省 ~10 个小数组 = 巨大 GC 减负。**单项最大加速：2.4×**。

**Python 版**：不需要（Python 的 tuple 分配已经够轻）。

---

### 23. 快速 ko 检测（内联 4 邻）✅ JS + Python 都使用

**描述**：判断"自方为单子"从 `own.group.length === 1` 改为直接检查 4 个邻位是否有己方棋子。避免为 ko 检测专门做一次 `groupAndLibs`。

**为什么用**：ko 检测在每次落子时执行。避免一次完整的 flood fill（只为知道 group 大小是否为 1）。

---

### 24. 快速 _terminal 内联 ✅ JS 版使用

**描述**：把 `getTargetGroup + countGroupRealEyes` 合并到 `_terminal` 函数内部，用专用 scratch 数组，无中间对象分配。

**为什么用（JS）**：消除 `getTargetGroup` 返回的对象 + `Uint8Array(100)` 分配。

**为什么 Python 版不用**：Python 版优先可读性，保持函数分离。PyPy 的 JIT 已经能很好地内联。

---

## 五、数据结构

### 25. `__slots__` ✅ Python 版使用

**描述**：在 `Board` 和 `UndoInfo` 类上声明 `__slots__`，告诉 Python 不使用实例 `__dict__`。

**为什么用**：减少每个实例约 200 字节的 dict 开销，属性访问略快。对频繁创建/销毁的 UndoInfo 尤其有效。

---

### 26. bytearray visited ✅ Python 版使用

**描述**：`group_and_libs` 中使用 `bytearray(size * size)` 做 visited 标志（代替 `set()`）。

**为什么用**：Python `set.add / set.__contains__` 每次要哈希 + 查表。`bytearray` 直接索引 O(1)，且分配固定 100 字节（远小于 set 的 ~2KB）。

---

### 27. 扁平索引替代 (x,y) 元组 ✅ Python 版部分使用

**描述**：在热路径中用 `i = y * size + x` 单整数替代 `(x, y)` 元组。减少 tuple 创建和解包。

**为什么用**：Python tuple 创建是 ~50ns 的开销。在百万次循环中累积显著。

---

## 六、运行时优化

### 28. PyPy ✅ 推荐使用

**描述**：Python 的 JIT（Just-In-Time）编译解释器。对纯 Python 代码自动编译热路径为机器码。

**为什么用**：零代码改动，`pypy3 backend/server.py` 即可。实测 **4.3× 加速**（CPython 14.5s → PyPy 3.4s）。

---

### 29. try/finally 保护 play/undo ✅ 当前使用

**描述**：df-pn 的 `_mid` 递归中，`_play_kid` 后接 `try: _mid(...) finally: _undo_kid()`。

**为什么用**：修复一个严重 bug——`_Timeout` 异常穿透递归栈时，所有层级的 `_undo_kid` 被跳过，棋盘处于脏状态。加 `try/finally` 保证无论是否超时，undo 一定执行。

---

## 七、界面与交互

### 30. 自动目标选择 ⛔ 已弃用

**描述**：`selectTarget()` 用启发式排序（区内子数 → 气少 → 眼少）自动选目标群。

**为什么用**：用户不必手动点选，系统自动推荐。

**为什么弃用**：启发式总有反例。用户在某道题上发现自动选错了（选了不想要的目标）。改为**强制手动点选**——严格工具不能猜。

---

### 31. "最大收益"目标排序 ⛔ 已弃用

**描述**：`selectTarget` 的排序改为"子数最多优先"（攻方吃最大块，防方保最大块）。

**为什么用**：用户请求"落子要设法获得最大收益"。

**为什么弃用**：用户随后改变需求，要求特定题目的特定首手。"最大收益"选出的目标不符合预期。最终删除所有自动选择，只保留手动。

---

### 32. 手动多目标点选 ✅ 当前使用

**描述**：用户在棋盘上点击棋子，白子 = 杀目标（红色高亮），黑子 = 守目标（蓝色高亮），再点取消。"确认目标"按钮锁定。

**为什么用**：用户需要"同时杀两块白棋 + 保住一块黑棋"的复合场景。手动选择消除所有启发式歧义。

---

### 33. 落子序号 ✅ 当前使用

**描述**：每步落子在棋子上叠加粗体序号。使用持久计数器（幸存子保留原序号）+ 被提子自动移除。

**为什么用**：用户要求"按照步数，在棋子上标上序号"，方便阅读对局记录。

---

### 34. 决策日志 ✅ 当前使用

**描述**：侧栏实时显示每手的攻/防角色、必胜/顽抗/试探标签、搜索元数据（节点、耗时、pn/dn）。

**为什么用**：用户要求"把所有的决策依据和落子依据都显示在网页上"。

---

### 35. 复杂度警告 ✅ 当前使用

**描述**：在目标标签下方显示"区域 X 格 · 空位 Y"，空位 > 15 显示红色警告"大概率 UNPROVEN，建议缩小区域"。

**为什么用**：37 格 / 20 空位的区域 df-pn 根本证不完（5 分钟 / 562 万节点仍 UNPROVEN）。用户需要直观看到"该缩区了"的信号。

---

## 十、一句话总结

**从 minimax 到 df-pn 是"正确性"的飞跃；从朴素实现到多轮优化是"速度"的飞跃。两者都不可或缺。** 当前的 Python + PyPy 方案在 ≤12 空位的区域内能在秒级给出数学上严格的最优解，同时保持了清晰可读的源码。
